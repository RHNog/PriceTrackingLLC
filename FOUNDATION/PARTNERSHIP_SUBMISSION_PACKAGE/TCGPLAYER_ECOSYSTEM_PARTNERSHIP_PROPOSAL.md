## Executive Summary

> Project Phronesis can strengthen the TCGplayer ecosystem by helping marketplace participants arrive with more context, more confidence, and clearer intent.

Project Phronesis is the internal engineering initiative responsible for developing an evidence-driven decision intelligence platform for collectible markets.

Project Phronesis is the engineering identity, not necessarily the future customer-facing product name. The commercial product name has intentionally not yet been chosen.

The project is inspired by the Greek concept of phronesis: practical wisdom, or the disciplined ability to make sound decisions under real-world conditions.

## Why Project Phronesis Exists

> The platform is not designed merely to display market data. It exists to help users make better decisions.

Project Phronesis applies practical wisdom to collectible-market decisions: evidence before conclusions, disciplined judgment before action, and explainable intelligence instead of opaque claims.

When users understand an asset more clearly before they buy, sell, restock, or monitor it, the marketplace can benefit from more informed participation.

**Operating Principles**

| Principle | Meaning |
| --- | --- |
| Practical Wisdom | Sound decisions under real-world conditions. |
| Evidence Before Conclusions | Market evidence should support claims before users act. |
| Reason Before Action | Users should understand why an asset may or may not be attractive. |
| Marketplace Enablement | Better decisions should strengthen trusted marketplaces. |

## Responsible Platform Philosophy

> Responsible integration means respecting partner infrastructure.

Project Phronesis has been intentionally designed to respect marketplace infrastructure.

At a public level, the platform is designed to reuse previously generated observations where appropriate, avoid unnecessary repeated external requests, separate development activity from production activity, request external data only when there is a workflow reason to do so, and support responsible integrations with clear information boundaries.

**Responsible API Philosophy**

| Principle | Public Meaning |
| --- | --- |
| Reuse Prior Work | Repeated user activity should not automatically mean repeated external provider activity. |
| Request With Purpose | External data should be requested when there is a workflow reason. |
| Respect Infrastructure | Marketplace and provider systems should be treated as shared infrastructure. |
| Keep Boundaries Clear | Business value can be discussed without exposing proprietary internals. |

This proposal does not disclose proprietary implementation details, internal reasoning systems, sensitive data assets, evaluation methods, or competitive algorithms.
