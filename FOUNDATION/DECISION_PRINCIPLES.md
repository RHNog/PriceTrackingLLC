# Decision Principles

These principles govern Project Phronesis decisions when the answer is not obvious.

They should be used before architecture, product, business, partnership, brand, or communication decisions become implementation.

## Evidence Before Conclusions

Do not produce conclusions before evidence is gathered, classified, validated, and understood.

Missing evidence should reduce confidence. It should not become negative evidence and should not be hidden.

## Explainability Over Opacity

If a user, engineer, or partner cannot understand why a claim exists, the claim is not ready.

The platform may use models, scores, and abstractions, but outputs must remain inspectable and defensible.

## Judgment Before Automation

Automation should help users decide. It should not erase human responsibility.

When automation is introduced, explain what it does, what evidence it uses, what it does not know, and where the user remains responsible.

## Platforms Over Features

Prefer capabilities that strengthen the shared platform.

An isolated feature may be useful, but it should not create duplicated logic, bypass core boundaries, or weaken the architecture's ability to support future workflows.

## Long-Term Thinking

Choose the path that future contributors can maintain, explain, and extend.

Speed matters, but not at the cost of hidden coupling, stale documentation, or unsupported claims.

## Responsible Partnerships

Partnerships should respect provider and marketplace infrastructure.

Project Phronesis should create mutual value, reduce unnecessary waste where possible, and avoid implying access, endorsement, or outcomes that do not yet exist.

## Marketplace Enablement

The platform should strengthen trusted marketplaces by creating better-informed participants.

It should not become a marketplace replacement unless a future strategic decision explicitly changes the business model and Foundation.

## Trust Before Growth

Do not trade credibility for speed.

If a claim cannot be defended, remove it. If a workflow cannot support an asset, say so. If evidence is incomplete, expose the limitation.

## Founder-Absent Decision Test

When the founders are not present, ask:

1. Does this preserve evidence before conclusions?
2. Does this make reasoning more explainable?
3. Does this support judgment rather than replace it?
4. Does this strengthen the platform rather than create an isolated shortcut?
5. Does this protect long-term trust?
6. Does this respect partners, providers, and marketplaces?
7. Does this remain honest about current capability?

If the answer is no, the decision should be revised or explicitly documented as an exception.
